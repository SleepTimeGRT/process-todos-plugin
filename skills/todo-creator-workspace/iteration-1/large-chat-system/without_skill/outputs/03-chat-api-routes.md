# Implement REST API routes for chat

The WebSocket layer handles real-time event delivery, but the REST API is responsible for initial data loading — fetching conversation lists, conversation history, and creating new conversations. These endpoints are called by the frontend on page load and navigation, before the WebSocket connection takes over for live updates.

- [ ] Create `src/app/api/conversations/route.ts`. Implement `GET` to return all conversations the authenticated user is a member of, ordered by most-recent message `updatedAt` desc. Each conversation object should include: `id`, `type`, `name`, `updatedAt`, the last message preview (`{ content, senderId, createdAt }`), member count, and the current user's `lastReadAt`. Implement `POST` to create a new conversation. Accept body `{ type: 'DIRECT' | 'GROUP', memberIds: string[], name?: string }`. For `DIRECT` conversations, check if one already exists between the two users and return it instead of creating a duplicate. Return 401 if unauthenticated.
- [ ] Create `src/app/api/conversations/[conversationId]/route.ts`. Implement `GET` to return conversation metadata plus its members `[{ id, name, image }]`. Return 403 if the authenticated user is not a member.
- [ ] Create `src/app/api/conversations/[conversationId]/messages/route.ts`. Implement `GET` to return paginated messages (query param `?cursor=<messageId>&limit=50`). Each message includes `id`, `content`, `createdAt`, `senderId`, `sender { name, image }`, and `readBy: [{ userId, readAt }]` from `MessageReadReceipt`. Return oldest-to-newest within the page. Implement `POST` to persist a message sent via the REST fallback (same write logic as the WebSocket `chat:send` handler — extract into a shared `src/lib/chat/createMessage.ts` helper so both paths share it).
- [ ] Create `src/lib/chat/createMessage.ts` that accepts `{ conversationId, senderId, content }`, validates membership, inserts the `Message`, and returns the `MessageWithSender` payload. This is the shared helper used by both the WebSocket handler and the REST `POST /messages` endpoint.
- [ ] Add input validation using `zod` for all `POST` request bodies. Define schemas in `src/lib/chat/schemas.ts`. Reuse these schemas on the frontend for form validation.
- [ ] Protect all routes with the existing auth session check. Return structured JSON errors `{ error: string, code: string }` with appropriate HTTP status codes (401, 403, 404, 422).
